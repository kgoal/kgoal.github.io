Activity 知识点

生命周期

onCreate

整个 Activity 的入口，生命周期内只执行一次。在这个生命周期内，需要通过 setCotentView 方法指定要显示的 View

onStart

在 Activity 展示到前台前，执行的操作

onResume

在这个生命周期时需要注意的是，按照上图的流程，是有可能会出现 onPause 生命周期到 onResume生命周期的。所以此时也会再次调用 onResume 方法

想要实现上述场景有两种方式，一种是通过一个半透明的 Activity 来遮挡现在的 Activity，具体查看 重现onPause到onResume生命周期；

另一种方式则是官方文档中提及的多窗口模式，在该模式下，如果点击另一个窗口，则其他的窗口则会处于 onPause 状态；

onPasue

在用户离开当前 Activity 时触发，也表明当前 Activity 不再处于前台（不意味着 Activity 不可见，参考上述例子）

该生命周期十分短暂，不推荐做耗时的操作

不像 onStart 和 onCreate， onPasue 执行完之后，Activity 不一定会发生变化，可能会处于 pause 状态，直到当前 Activity 对用户不可见，然后进入 stop 状态或者是 Activity 重新活动，进入 resume 状态

onStop

此时 Activity 已经不可见了，但是 Activity 还是被保存在内存中。内存中的 Activity 可能会因为内存不足而被系统释放。

在该回调方法可以执行耗时的操作

根据生命周期图，可以从 stop 状态进入 onCreate,onRestart 一集 onDestroy

onDestroy

可以通过 isFinishing() 方法来判断 Activity 是否被销毁，因为有可能是因为 Activity 的配置改变而执行的 onDestroy 方法，在此之后，系统会立刻创建一个新的 Activity。

多 Activity 的生命周期

当从 A 跳转至 B 时：

  1. A -> onPasue
  2. B -> onCreate -> onStart -> onResume
  3. A -> onStop

LifecycleObserver

这是官方提供的一个组件，能够对 Activity 的生命周期进行监听，会触发事件，订阅者可以收到事件响应。

同样的逻辑，官方更加的推荐使用 LifecycleObserver 的方式来实现，因为这样可以实现代码的服用，避免在多个 Activit 中编写 onResume 的相同逻辑

状态保存

对于 Activity，其有可能会被系统销毁，也有可能按照用户的想法进行销毁。对于前者，需要将当前的状态保存下来。

默认的 onSaveInstanceState 方法只会将 Activity 的视图状态保留下来，如果需要保存自定义的变量，需要重写该方法来完成。

该方法是由系统自动调用的，但如是 Activity 是因为调用了 finish() 方法而被关闭，该方法是不会被执行的。同理，按下返回键也是如此

也不推荐在 onSaveInstanceState 方法中对复杂数据进行保存，因为这会进行序列化和反序列化的操作。

至于保存起来的状态，可以在 onCreate 或者是 onRestoreInstanceState 方法中拿到。

app 之间交互

通过 Intent 来启动其他 APP，通过 Action 和 uri 两个值来决定使用哪种 app

其中选择器的作用在于传入的第二个参数会作为弹出选择框的标题，并且如果设置了选择器，在没有可用 app 时，也会弹出选择框，并显示没有可用的 app

除了使用 startActivity，也可以使用 startActivityForResult ，处理返回的结果。每个 app 处理的结果都不相同，需要根据约定好的方式来解析数据

通过在 AndroidManifest 文件中，为 Activity 设置好 <Intent-filter> 标签以及 action，category 和 data 属性，能够将 activity 提供给其他 app 使用。
