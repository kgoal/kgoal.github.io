可进行操作的通知

可进行操作的通知比较常见，常见的如下：

  • 点击通知后打开 app 并跳转至指定页面
  • 点击通知提供的按钮，执行对应的操作
  • 点击回复按钮，能够在通知中回复消息

进度条通知

  1. 相对于操作来说更加的方便，只需要在 NotificationCompat.Builder 方法中调用 setProgress(progressMax, progress, false) 即可。第一个参数表示进度最大值，第二个是当前进度，第三个表示该进度是否是假的。
  2. 然后只需要在后续重复调用该方法，传入的 progress 改变，那么进度条也会随之改变。
  3. 最后，在进度完成时，继续调用该方法，不过将 progress 和 progressMax 都等于0，这样进度条就会消失。（所以这时候可以设置 SetContentText 提示用户下载完成）
  4. 如果第三个参数是 true 时，进度条会不停的动，想要在进度完成后，关闭进度条，需要按照上述的停止方法并且将参数设置为 false
  5. 在进度条存在时，ContentText 的位置会被移动至进度条的右上角，所以可以用来做进度百分比显示

悬挂式

上面有提到的悬挂式通知，其效果如下，通知会直接显示在页面上

悬挂式通知有两种，一种会自动缩起，一种会一直停留在页面上。推荐使用第一种。

第一种的用法很简单，只需要将 setPriority 中的优先级参数设置为高于 PRIORITY_DEFAULT ，以及对应的 channel 的重要程度，也设置高于 IMPORTANCE_DEFAULT。

这样通知就可以短暂的悬挂显示在屏幕上。

可能由于是 Android 系统的原因，不能频繁的发送悬挂式通知，需要过段时间，才能够再发送弹出的通知

第二种则是通过 setFullScreenIntent() 来设置，接收两个参数，第一个位 pendingIntent，用于点击后的操作，理论上和 setContentIntent 一致；第二个参数是 highpriority ，布尔值,用于决定在当前页面上已经有悬挂式通知时，是否还显示该通知。

这种方式生成的通知，不会自动缩起，会一直在显示着，并且，传入的 intent 还有可能会自动执行。

问题描述： 在第一次发送该通知后，如果点击执行了这个 intent，后续的通知，都会直接去执行，而不再是弹出悬挂式通知。
问题原因分析（未检验过）：由于只是 demo，使用的 intent 和消息 id 是一样的，所以导致判断是同一条消息。从而自动执行
